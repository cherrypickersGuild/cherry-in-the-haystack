# 백엔드 기사 가공 파이프라인 기획서

## 개요
기사 수집부터 AI 가공, 스냅샷 생성까지의 백엔드 파이프라인을 단계별로 구현한다.
각 단계는 독립적으로 테스트 가능하도록 목(mock) SQL과 서비스 코드를 분리한다.

기존 `seed-mock-data.sql`은 모든 컬럼을 한 번에 채워 단계별 테스트가 불가능했음.
에이전트 write spec(`apps/docs/agent-column-write-spec-2026-04-07.md`)에 따라 파이프라인을 6단계로 분리:

1. 참조 데이터 → 2. Collector가 article_raw 수집 → 3. 백엔드가 user_article_state 생성 → 4. 백엔드가 PENDING ai_state 생성 → 5. Agent가 agent_json_raw 작성 → 6. Worker가 파싱하여 정규 컬럼 투영

각 단계별로 SQL을 분리하고, SQL 적용 후 해당 단계에 필요한 백엔드 서비스를 작성/테스트하는 사이클을 반복한다.

> 개인화 파이프라인(`pipeline_user`)은 현재 구현 범위 밖. 코드는 보존하되 이 기획서에서는 다루지 않음.

## 시스템 유저 (공통 피드용)

개인화 없는 공통(public) user_article_state의 user_id로 **시스템 유저**를 사용한다.

| 항목 | 값 |
|------|-----|
| ID | `00000000-0000-0000-0000-000000000000` |
| name | `__SYSTEM__` |
| email | `system@cherry.internal` |
| seed 파일 | `apps/docs/seed_data/system-user.sql` |

> DDL constraint `chk_app_user_id_not_reserved` 제거됨 (nil UUID 허용).
> 실제 로그인 불가. magic_token 없음.

## Stage 1 — article_raw INSERT (External Collector 시뮬레이션) ✅ 완료

### SQL 내용
100건 INSERT. **Collector가 채우는 컬럼만** 사용:

| 채우는 컬럼 | 설명 |
|-------------|------|
| `id` | `0195f300-a{seq:03d}-7000-8000-000000000001` |
| `source_id` | 10개 source 중 하나 |
| `title` | 기사 제목 (기존 seed 데이터) |
| `url` | `https://example.com/article-{seq}` |
| `published_at` | 2026-02-10 ~ 2026-04-06 |
| `representative_key` | `seed-{seq}-{md5(title)}` |
| `language` | 'en' 또는 'ko' (source 기반) |

| NULL로 남기는 컬럼 | 이유 |
|---------------------|------|
| `content_raw` | RSS 단계에서는 전문 미수집 |
| `guid`, `normalized_url`, `canonical_url` | 선택적 |
| `author`, `image_url` | 선택적 |
| `external_storage_key` | S3 미사용 |
| `content_hash` | 전문 없으므로 NULL |

자동 생성: `url_hash`, `representative_key_hash` (GENERATED ALWAYS)
자동 기본값: `fetched_at=NOW()`, `storage_state='ACTIVE'`

### SQL 적용 후 빌드할 백엔드 ✅ 완료
**`ArticleIngestionService`** (`apps/api/src/modules/pipeline/article-ingestion.service.ts`)
- 역할: article_raw 1건 → 시스템 유저용 user_article_state 1건 생성
- 입력: `articleRawId` (UUID)
- 출력: `user_article_state` INSERT (user_id = 시스템 유저)
- 중복: partial unique index(`user_id, article_raw_id WHERE revoked_at IS NULL`) → 존재 시 스킵
- API: `POST /api/pipeline/ingest`

### 검증 쿼리
```sql
SELECT count(*) FROM content.article_raw WHERE representative_key LIKE 'seed-%';  -- 100
SELECT count(*) FROM content.article_raw WHERE content_raw IS NULL;               -- 100
SELECT count(*) FROM content.user_article_state;                                   -- 0 (아직)
```

### 테스트 시나리오
```
POST /api/pipeline/ingest
{ "articleRawId": "0195f300-a001-7000-8000-000000000001" }
→ { "created": true }   # 재호출 시 false
```

## Stage 2 — user_article_state INSERT (Backend 시뮬레이션)

### SQL 내용
100건 INSERT. **Backend가 Stage 1 후 생성하는 bare-bones 레코드**:

| 채우는 컬럼 | 값 |
|-------------|-----|
| `id` | `0195f300-b{seq:03d}-7000-8000-000000000001` |
| `user_id` | `00000000-0000-0000-0000-000000000000` (시스템 유저) |
| `article_raw_id` | Stage 1의 article_raw.id |
| `discovered_at` | `published_at + INTERVAL '1 hour'` |

| 기본값/NULL 컬럼 | 값 | 이유 |
|-------------------|-----|------|
| `representative_entity_id` | **NULL** | Worker가 Stage 5에서 설정 |
| `side_category_id` | **NULL** | Worker가 Stage 5에서 설정 |
| `impact_score` | **0** (DEFAULT) | Worker가 Stage 5에서 계산 |
| `is_high_impact` | **FALSE** (DEFAULT) | Worker가 Stage 5에서 계산 |
| `is_hidden` | **FALSE** (DEFAULT) | |
| `meta_json` | **NULL** | |

⚠️ **기존 seed와의 핵심 차이**: 기존 seed는 이 시점에 `representative_entity_id`, `impact_score`를 이미 채웠으나, 실제 파이프라인에서는 AI 처리 전이므로 알 수 없음.

### SQL 적용 후 빌드할 백엔드
**`AiStatePreGenService`** (`apps/api/src/modules/pipeline/ai-state-pregen.service.ts`)
- 역할: user_article_state 생성 후 PENDING user_article_ai_state 레코드 생성
- 입력: `userArticleStateId`, `userId`
- 출력: `user_article_ai_state` INSERT (ai_status='PENDING')
- API: `POST /api/pipeline/pregen-ai-state`

### 검증 쿼리
```sql
SELECT count(*) FROM content.user_article_state;                                       -- 100
SELECT count(*) FROM content.user_article_state WHERE representative_entity_id IS NULL; -- 100 (전부)
SELECT count(*) FROM content.user_article_state WHERE impact_score = 0;                -- 100 (전부)
SELECT count(*) FROM content.user_article_ai_state;                                     -- 0 (아직)
```

## Stage 3 — ai_state PENDING (Backend 시뮬레이션)

### SQL 내용
100건 INSERT. **AI 처리 전 PENDING 레코드**:

| 채우는 컬럼 | 값 |
|-------------|-----|
| `id` | `0195f300-c{seq:03d}-7000-8000-000000000001` |
| `user_id` | `00000000-0000-0000-0000-000000000000` (시스템 유저) |
| `user_article_state_id` | Stage 2의 id |
| `ai_status` | `'PENDING'` |

| NULL인 컬럼 (전부) | 이유 |
|---------------------|------|
| `representative_entity_*` (5개) | Agent가 아직 분석 안 함 |
| `ai_summary`, `ai_score` | Agent가 아직 분석 안 함 |
| `ai_classification_json` ~ `ai_structured_extraction_json` | Agent가 아직 분석 안 함 |
| `agent_json_raw` | Agent가 아직 작성 안 함 |
| `ai_model_name`, `ai_processed_at` | Agent가 아직 작성 안 함 |
| `prompt_template_version_id`, `run_log_id` | 선택적 |

✅ DDL constraint `chk_user_article_ai_state_entity_snapshot_presence`: representative_* 5개 전부 NULL → ALL NULL 조건 만족

### SQL 적용 후 빌드할 백엔드
**`AgentOrchestratorService`** (`apps/api/src/modules/pipeline/agent-orchestrator.service.ts`)
- 역할: PENDING 레코드 조회 → 프롬프트 컨텍스트 조립 → Agent 호출
- 컨텍스트 조립 내용:
  - `user_article_state_id`, `user_id`
  - article: `title`, `url`, `content_raw`, `published_at`
  - `allowed_entities`: tracked_entity + placement JOIN → `{id, page, category_id, category_name, name}` 리스트
- API: `GET /api/pipeline/pending-batch`

### 검증 쿼리
```sql
SELECT count(*) FROM content.user_article_ai_state WHERE ai_status = 'PENDING';      -- 100
SELECT count(*) FROM content.user_article_ai_state WHERE agent_json_raw IS NOT NULL; -- 0
SELECT count(*) FROM content.user_article_ai_state WHERE ai_summary IS NOT NULL;     -- 0
```

## Stage 4 — agent_json_raw UPDATE (Agent 시뮬레이션)

### SQL 내용
100건 **UPDATE** (INSERT가 아님). 기존 PENDING 레코드에 agent_json_raw만 설정:

| UPDATE 컬럼 | 값 |
|-------------|-----|
| `agent_json_raw` | 전체 JSON v0.3 페이로드 |
| `ai_model_name` | `'claude-3.7-sonnet'` |
| `ai_processed_at` | `published_at + INTERVAL '2 hours'` |

| 변경하지 않는 컬럼 (Agent 금지) | 이유 |
|--------------------------------|------|
| `ai_status` | **여전히 PENDING** (Worker가 SUCCESS로 변경) |
| `representative_entity_*` (5개) | Backend Worker만 작성 가능 |
| `ai_summary`, `ai_score` | Backend Worker만 작성 가능 |
| `ai_classification_json` ~ `ai_structured_extraction_json` | Backend Worker만 작성 가능 |

### agent_json_raw 구조 (v0.3)
```json
{
  "idempotency_key": "uas:{lower(user_article_state_id)}",
  "version": "0.3",
  "representative_entity": {
    "id": "...", "page": "MODEL_UPDATES",
    "category_id": "...", "category_name": "OpenAI Family",
    "name": "GPT-5.4"
  },
  "ai_summary": "...",
  "ai_score": 4,
  "ai_classification_json": { "final_path": {}, "candidates": [], "decision_reason": "..." },
  "ai_tags_json": [{ "kind": "TAG", "value": "..." }],
  "ai_snippets_json": { "why_it_matters": "...", "key_points": [], "risk_notes": [] },
  "ai_evidence_json": { "evidence_items": [{ "kind": "quote", "text": "...", "url": "...", "source_name": "...", "published_at": "..." }] },
  "ai_structured_extraction_json": { "source": { "name": "...", "type": "RSS" }, "review": { "type": "Announcement", "reviewer": null, "comment": null } }
}
```

⚠️ CASE_STUDIES 페이지 기사의 경우 `ai_structured_extraction_json.review.type`에 "Case Study" 또는 "Applied Research" 포함 → Stage 5에서 side_category 결정에 사용

### SQL 적용 후 빌드할 백엔드 ★ 핵심
**`AgentJsonParserService`** (`apps/api/src/modules/pipeline/agent-json-parser.service.ts`)
- 역할: agent_json_raw 파싱 → 엔터티 검증 → 정규 컬럼 투영
- 상세:
  1. `agent_json_raw->'representative_entity'->>'id'`로 tracked_entity 조회
  2. tracked_entity_placement에서 (entity_id, page) 매칭 → canonical category 확인
  3. 이름 불일치 시 canonical 값으로 덮어쓰기
  4. 검증 실패 시 `ai_status = 'FAILED'`
  5. 성공 시 정규 컬럼 모두 채우고 `ai_status = 'SUCCESS'`
- API: `POST /api/pipeline/parse-batch`

### 검증 쿼리
```sql
SELECT count(*) FROM content.user_article_ai_state WHERE agent_json_raw IS NOT NULL;           -- 100
SELECT count(*) FROM content.user_article_ai_state WHERE ai_status = 'PENDING';                -- 100 (아직)
SELECT count(*) FROM content.user_article_ai_state WHERE ai_summary IS NOT NULL;               -- 0 (아직)
SELECT count(*) FROM content.user_article_ai_state WHERE representative_entity_id IS NOT NULL; -- 0 (아직)
```

## Stage 5 — Worker Parse & Project (Worker 시뮬레이션)

### SQL Part A: user_article_ai_state UPDATE

```sql
UPDATE content.user_article_ai_state aai
SET
  representative_entity_id = te.id,
  representative_entity_page = tep.entity_page,
  representative_entity_category_id = tep.entity_category_id,
  representative_entity_category_name = ec.name,
  representative_entity_name = te.name,
  ai_summary = aai.agent_json_raw->>'ai_summary',
  ai_score = (aai.agent_json_raw->>'ai_score')::SMALLINT,
  ai_classification_json = aai.agent_json_raw->'ai_classification_json',
  ai_tags_json = aai.agent_json_raw->'ai_tags_json',
  ai_snippets_json = aai.agent_json_raw->'ai_snippets_json',
  ai_evidence_json = aai.agent_json_raw->'ai_evidence_json',
  ai_structured_extraction_json = aai.agent_json_raw->'ai_structured_extraction_json',
  ai_status = 'SUCCESS'
FROM content.tracked_entity te
JOIN content.tracked_entity_placement tep ON ...
JOIN content.entity_category ec ON ...
WHERE te.id = (aai.agent_json_raw->'representative_entity'->>'id')::UUID
  AND aai.ai_status = 'PENDING'
  AND aai.agent_json_raw IS NOT NULL;
```

### SQL Part B: user_article_state UPDATE

```sql
UPDATE content.user_article_state uas
SET
  representative_entity_id = aai.representative_entity_id,
  side_category_id = CASE
    WHEN ... = 'Case Study' THEN (SELECT id FROM content.side_category WHERE code = 'CASE_STUDY' ...)
    WHEN ... = 'Applied Research' THEN (SELECT id FROM content.side_category WHERE code = 'APPLIED_RESEARCH' ...)
    ELSE NULL END,
  impact_score = CASE WHEN aai.ai_score = 5 THEN 80+... ... END,
  is_high_impact = (aai.ai_score >= 4)
FROM content.user_article_ai_state aai
WHERE uas.id = aai.user_article_state_id AND aai.ai_status = 'SUCCESS';
```

### SQL 적용 후 빌드할 백엔드
**`SnapshotBuilderService`** (`apps/api/src/modules/pipeline/snapshot-builder.service.ts`)
- API: `POST /api/pipeline/build-snapshot`

### 검증 쿼리
```sql
SELECT count(*) FROM content.user_article_ai_state WHERE ai_status = 'SUCCESS';               -- 100
SELECT count(*) FROM content.user_article_ai_state WHERE ai_summary IS NOT NULL;              -- 100
SELECT count(*) FROM content.user_article_ai_state WHERE representative_entity_id IS NOT NULL; -- 100
SELECT count(*) FROM content.user_article_state WHERE representative_entity_id IS NOT NULL;    -- 100
SELECT count(*) FROM content.user_article_state WHERE impact_score > 0;                        -- 100
SELECT count(*) FROM content.user_article_state WHERE side_category_id IS NOT NULL;            -- ~15 (CASE_STUDIES)
SELECT count(*) FROM content.user_article_state WHERE is_high_impact = TRUE;                   -- ~60
```
