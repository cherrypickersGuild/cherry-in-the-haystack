## 🚨 실수 방지 — 이전 세션에서 발견한 함정

### 함정 1: MCP tool API 이름
- ❌ `server.registerTool(...)` — 존재 X
- ✅ `server.tool(name, description, schema, handler)` — 실제 API

확인: `apps/agent-package/lib/mcp-tools.js:41` 기존 예시

### 함정 2: MCP tool 배치 위치
- 새 tool은 반드시 **`function registerTools(server, apiKey, baseUrl) { ... }` 함수 내부**에 배치
- 기존 `follow_concept` 등록 블록 바로 아래, 같은 들여쓰기

### 함정 3: 엔드포인트 경로
- ❌ `/api/v1/a2a` — 잘못
- ✅ `/api/v1/kaas/a2a` — 맞음
- NestJS `app.setGlobalPrefix('api')` + controller `@Controller('v1/kaas/a2a')` 조합

### 함정 4: `PUBLIC_API_URL` 환경변수
- Agent Card의 `url` 필드에 사용됨
- **`apps/api/.env`에 반드시 추가**해야 함:
  ```
  PUBLIC_API_URL=http://localhost:4000    # 로컬
  PUBLIC_API_URL=https://api.solteti.site # AWS (실제 도메인 확인 필수)
  ```
- 누락 시 Agent Card url 필드가 `/api/v1/kaas/a2a` (도메인 없이) 나옴

### 함정 5: `agentService.findAll` 없음
- 기존 agent service에는 `findByUserId`, `findByApiKey`, `findById`만 있음
- 에이전트 목록이 필요하면 **`KaasA2aService`에 `listActiveAgents()` 직접 추가** (기획서에 이미 반영됨)

### 함정 6: Cherry Console `agents` stale closure
- `kaas-console.tsx`의 `useEffect` deps는 `[currentAgent?.id, currentApiKey]`만
- 여기서 `agents` 변수 참조 시 초기값 고정됨
- **데모용 권장: 필터 제거 후 전체 이벤트 표시** (기획서에 필터 제거 코드 있음)

### 함정 7: Windows 경로 파싱
- 기존 코드에서 Linux/Mac (`/`) 과 Windows (`\`) 경로 split 구분 필요
- `split(/[\\/]/)` 정규식 사용 (이미 수정됨, 참고)

### 함정 8: 에이전트 번들 배포 3곳 동기화 (기존 패턴)
빌드 후 아래 3곳 확인 (프로젝트 기존 운영 패턴):
1. `apps/agent-package/dist/cherry-agent.js` (원본)
2. `apps/web/public/cherry-agent.js` (웹에서 다운로드용, 있을 경우)
3. npm 퍼블리시된 위치 (있을 경우)

## 🎭 Cherry Console 특이사항 (가장 중요)

### 왜 중요한가
**요구사항**: "결국 체리 콘솔에 나와야 한다" — 유저가 강조
→ A2A task 생성/응답이 **웹 Cherry Console 플로팅 창에 실시간 🔀 A2A 박스**로 표시되어야 함.

### 기존 Console의 WebSocket 구조
- 파일: `apps/web/components/cherry/kaas-console.tsx`
- 위치: 약 681번 줄 `useEffect` 블록
- 연결: `io(\`${base}/kaas\`, { auth: { api_key, role: "user" } })`
- 관리: `roomSocketRef.current`
- 기존 이벤트: `room_message`, `agent_report_pushed`
- **추가할 이벤트**: `a2a_task_event`

### 서버 측 브로드캐스트
- `KaasA2aService.broadcastToConsoles()` 메서드 (기획서 Step 2에 구현)
- `this.wsGateway.server.emit('a2a_task_event', { type, task, from_agent, to_agent })`
- `@WebSocketServer() server: Server` 는 public이라 외부 service에서 접근 가능

### 메시지 타입 확장
Message union에 `role: "a2a"` 추가:
```typescript
| {
    role: "a2a"
    eventType: "task_created" | "task_responded" | "task_canceled" | "task_completed"
    fromName: string; toName: string
    fromId: string; toId: string
    text: string; taskId: string; status: string; ts: string
  }
```

### 렌더링
보라색 박스 (`bg-purple-50 border-purple-200`) + `🔀 A2A` 아이콘 + `fromName → toName` + 본문 + status.

### 검증 순서
1. curl로 수동 task 생성 → Console에 박스 뜨나? (T9.2)
2. 실제 에이전트 연결 상태에서 → 박스 뜨나? (T10.1)
3. 크로스 머신에서도 → 박스 뜨나? (T12.1)

## 📣 사용자 요구사항 정확히 반영

### 명시된 것
1. **Mac + Linux 크로스 머신** — 유저가 "리눅스 항상 접속 가능"이라 명시
2. **AWS 배포 환경** — Cherry는 AWS 도메인에 이미 배포됨
3. **A2A 표준 준수** — "표준대로 가는 게 좋지 않아?" 라고 동의, 커스텀 확장은 `x-cherry` prefix로
4. **NestJS 기존 KaasModule에 추가** — 새 모듈 X
5. **Cherry Console 실시간 표시** — 유저가 여러 번 강조한 핵심 요구
6. **기초 엔드포인트 먼저** — 유즈케이스(코인 거래 등)는 나중에 붙이는 식
7. **점진적 확장성** — task_type + JSONB input/output로 미래 확장 담보
8. **개발자 테스트** — 발표용 데모 완벽함보다 기능 검증 우선

### 명시 안 됨 (데모 시점까지 결정 필요)
- Linux Claude Code까지 띄워서 MCP 자연어 트리거 할지 (T12.3)
- Mac-Linux 관계자/역할 시나리오 (단순 ping/pong 또는 실제 업무)
- 구체적 task_type 활용 (message/signal/order 등)
- 경제 레이어 (credit 차감, escrow 등 — 나중)
